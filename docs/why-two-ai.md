# 为什么需要双AI？

> 单AI自问自答的致命缺陷，以及双AI分工如何解决。

---

## 单AI的问题

### 1. 幻觉无法自查

单AI在生成代码时，可能会：
- 幻觉出不存在的字段名（如 `relevance_score` 而实际是 `relevance`）
- 幻觉出不存在的方法名（如 `call_model()` 而实际是 `_call_api()`）
- 幻觉出不存在的文件路径

**问题**：AI 自己生成的代码，自己审查时会认为"这是对的"，因为它不会质疑自己的输出。

**双AI解决**：策略AI 审查执行AI的输出，对比 PROBE 阶段发现的实际字段名和方法名，发现不一致立即指出。

### 2. 方向偏离无法自纠

单AI在多轮对话中可能会：
- 忘记最初的需求，走向错误方向
- 在错误假设上堆代码，越改越错
- 看起来"代码能跑"，但底层逻辑是错的

**问题**：AI 没有"回头看"的机制，不会主动检查"我是不是走偏了"。

**双AI解决**：策略AI 在每个阶段审查执行AI的输出，判断"这是否符合最初的需求"，发现偏离立即纠正。

### 3. 一次改多个文件无法自控

单AI在执行复杂任务时可能会：
- 一次性修改多个文件，导致调用链断裂
- 改了A文件但忘了同步修改B文件
- 输出的代码片段互相矛盾

**问题**：AI 没有"一次只做一件事"的纪律，倾向于"一次性解决所有问题"。

**双AI解决**：工作流强制一次只改一个文件，策略AI 审查"是否只改了1个文件"，发现违规立即制止。

## 双AI的分工

### 策略AI（窗口A）

**职责**：审查方向，不碰代码

- 审查执行AI的输出是否完整
- 判断当前代码状态（全新/部分/已完成）
- 生成下一阶段的执行指令
- **绝不生成具体代码**

**适合的AI**：Kimi、ChatGPT、Claude（擅长分析和判断）

### 执行AI（窗口B）

**职责**：写代码，不判断方向

- 按照指令查代码、改代码、运行验证
- 输出状态标记（如 `[PROBE_DONE]`）
- **绝不自行决定下一步**

**适合的AI**：Mimo Code、Cursor、Copilot（擅长代码操作）

## 用户作为信息枢纽

用户在双AI架构中的角色是**信息枢纽**：

```text
┌─────────────┐          ┌─────────────┐
│   策略AI     │◄────────►│    用户      │◄────────►│   执行AI     │
│  （窗口A）   │  审查结果  │  （信息枢纽） │  指令/结果 │  （窗口B）    │
│  Kimi等      │◄────────►│   你         │◄────────►│   Mimo       │
└─────────────┘          └─────────────┘          └─────────────┘
```

用户的操作：
1. 把执行AI的输出复制给策略AI审查
2. 把策略AI的指令复制给执行AI执行
3. 确认或回退

**用户不需要懂代码**，只需要会复制粘贴和点确认。

---

## 类比

| 角色 | 职责 | 对应 |
|------|------|------|
| **导演** | 审查方向，判断"这场戏对不对" | 策略AI |
| **演员** | 按照剧本表演，不自己改剧本 | 执行AI |
| **场记** | 记录进度，传递信息，确认"过了" | 用户 |

- 导演不亲自演戏（策略AI不写代码）
- 演员不自己决定下一场演什么（执行AI不判断方向）
- 场记不需要会演戏（用户不需要懂代码）

---

## 什么时候不需要双AI？

- 任务非常简单（改一行代码、加一个字段）→ 直接用单AI
- 用户自己是程序员，能审查代码 → 单AI + 自己审查
- 任务有明确的规格文档 → 单AI + 规格约束

## 什么时候需要双AI？

- 用户不会编程，但想用AI改造项目
- 任务复杂，涉及多个文件和模块
- 之前用AI改崩过，想有安全保障
- 需要可重复、可回退的开发流程
