# 02 - DESIGN：冻结架构

> **阶段目标**：确认"改造后长什么样""怎么验证""失败了怎么回退"。
>
> 根据 PROBE 阶段的状态判断，选择对应分支。

---

## 分支A：状态A（全新）→ DESIGN-从零改造

### 执行AI提示词（发给窗口B）

```text
【阶段：DESIGN】只回答设计问题，不写代码，不执行命令。

基于你刚才查到的代码，本次改造目标：{{重复你的需求}}

当前代码状态（PROBE发现）：
- 关键词提取在 {{文件}} 的 {{方法}}，使用 {{旧实现方式}}
- 记忆域选择在 {{文件}} 的 {{方法}}，使用 {{旧实现方式}}
- 模型调用能力在 {{文件}} 的 {{方法}}
- 判断逻辑当前是 {{批量/逐条}}，在 {{文件}}

请回答：
1. 【数据流】改造后的完整数据流是什么？用 → 表示。标注新增/修改/删除的环节。
2. 【改造点】本次改造涉及哪几个文件？按依赖顺序排列（被依赖的先改）。
3. 【调用关系】新增/修改的模块由谁调用？输出给谁？画出调用关系。
4. 【回退策略】如果新逻辑失败，系统如何回退？回退路径是什么？
5. 【验证方案】如何证明改造生效？建议在哪几个位置添加临时日志/探针？
6. 【风险点】改造可能破坏哪些现有功能？如何规避？

回答清楚后，我回复"确认"你再写代码。
输出末尾加 [DESIGN_DONE]
```

### 等待执行AI返回

你会收到设计方案（6个答案 + `[DESIGN_DONE]`）。

把完整返回复制下来。

### 策略AI提示词（发给窗口A）

```text
【DESIGN审查任务】

请审查以下Mimo返回的DESIGN结果。

## Mimo的DESIGN返回：
{{Mimo的DESIGN返回}}

## 你的任务：

### 第一步：完整性审查
检查以下项目：
- [ ] 有完整的数据流图（包含 → 箭头，从用户输入到最终输出）
- [ ] 列出了具体改造文件（有文件路径）
- [ ] 有明确的修改顺序（先改哪个、后改哪个）
- [ ] 回退策略清晰（失败时调用哪个旧方法）
- [ ] 验证方案具体（不是"看结果"，而是"在X文件Y行加logger"）
- [ ] 列出了至少1个风险点
- [ ] 末尾有 [DESIGN_DONE] 标记

如果以上有任何一项未通过，输出【追问指令】。

### 第二步：输出下一阶段指令

如果审查通过，输出【CONTRACT-契约确认】指令（见 03-contract.md）。
```

## 分支B：状态B（部分）→ DESIGN-诊断

### 执行AI提示词（发给窗口B）

```text
【阶段：DESIGN-诊断】不修改代码，只设计排查方案。

PROBE发现：
- {{已存在的文件/方法}} 已存在
- {{调用链位置}} 已调用新方法
- 但用户运行时 {{具体现象，如"关键词仍是正则结果"}}

请设计排查方案，回答：
1. 【假设列表】{{功能}} 已存在但输出仍是 {{旧结果}}，列出所有可能原因，按概率排序。
2. 【验证点】为验证每个假设，需要在哪些位置加什么日志？具体到文件、方法、行号。
3. 【排查顺序】按概率从高到低列出排查步骤。
4. 【修复预判】最可能的问题是什么？修复涉及哪些文件？
5. 【验证方案】修复后如何验证？需要触发什么操作？预期看到什么日志？

不要写代码。
输出末尾加 [DESIGN_DONE]
```

### 等待执行AI返回

你会收到诊断方案。

把完整返回复制下来。

### 策略AI提示词（发给窗口A）

```text
【DESIGN-诊断审查任务】

请审查以下Mimo返回的诊断方案。

## Mimo的诊断方案：
{{Mimo的诊断方案}}

## 你的任务：

### 第一步：完整性审查
- [ ] 假设列表按概率排序
- [ ] 验证点具体到文件、方法、行号
- [ ] 排查顺序合理（高概率优先）
- [ ] 修复预判有依据
- [ ] 末尾有 [DESIGN_DONE] 标记

### 第二步：输出下一阶段指令

如果审查通过，输出【CODE-诊断修复】指令（加日志定位根因，见 04-code.md）。
```
