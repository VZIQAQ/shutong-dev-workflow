# 04 - CODE：单文件推进

> **阶段目标**：一次只改一个文件，改完一个等你确认再改下一个。
>
> 如果有多个文件要改，重复本阶段（文件1 → 文件2 → ... → 文件N）。

---

## 执行AI提示词（发给窗口B）

### 第1个文件

```text
【阶段：CODE-文件1】只改这一个文件，不要动其他文件。

【绝对铁律】
1. 用户没有说"确认"，绝不进入下一个文件
2. 绝不一次性修改多个文件
3. 每改完一个文件，必须 git add + git commit（提交信息写明改了什么）
4. 如果修改失败，必须 git checkout 回退到上一次提交
5. 禁止 git push --force、git reset --hard 等破坏性操作
6. 禁止一次性 git add . （必须指定具体文件）

{{新建/修改}} 文件：{{文件路径}}

改造目标：{{具体功能描述}}

当前项目上下文（PROBE发现）：
- 模型调用封装在 {{文件}} 的 {{方法}}
- 相关数据结构：{{数据对象}} 字段为 {{字段列表}}
- 旧方法 {{方法名}} 在 {{文件}} 保留备用

要求：
- 只输出变更部分的代码，用注释标明插入位置
- 如果新建文件，输出完整代码
- 关键路径必须加 [PROBE] 日志，格式：logger.info(f"[PROBE] 具体信息")
- 所有异常必须捕获，不能抛未处理异常
- 代码风格与项目现有代码保持一致
- 不要重启服务，不要运行测试，不要读取日志
- 只输出这个文件的代码

输出后加 [FILE_DONE: {{文件名}}]
等待我发"下一步"再继续。
```

## 等待执行AI返回

你会收到代码修改（`[FILE_DONE: 文件名]`）。

把完整返回复制下来。

---

## 策略AI提示词（发给窗口A）

把 `{{Mimo的代码修改}}` 替换为你复制的内容：

```text
【CODE审查任务】

请审查以下Mimo返回的代码修改。

## Mimo的代码修改：
{{Mimo的代码修改}}

## 你的任务：

### 第一步：合规性审查
- [ ] 只改了1个文件（输出中只有1个 [FILE_DONE]）
- [ ] 有 [PROBE] 日志（代码中有 logger.info(f"[PROBE] ...") ）
- [ ] 有异常捕获（有 try/except 或类似机制）
- [ ] 引用了PROBE中的方法（如 _call_api）
- [ ] 字段名与CONTRACT一致（没有幻觉字段名）

### 第二步：判断下一步

如果还有下一个文件要改，输出【CODE-文件N】指令。
如果是最后一个文件，输出【VERIFY】指令（见 05-verify.md）。
```

---

## 循环执行

- 如果策略AI说"还有下一个文件" → 复制【CODE-文件N】指令发给执行AI，重复本阶段
- 如果策略AI说"最后一个文件" → 复制【VERIFY】指令，进入 [05-verify.md](05-verify.md)
