# 07 - 异常处理速查表

> 当执行AI违规或卡住时，复制对应话术发给它。

---

## 违规现象 → 话术对照表

| 违规现象 | 你发给执行AI的话 |
|---------|----------------|
| 跳过探测直接写代码 | `停。回到PROBE阶段。先不要写代码，回答6个问题。` |
| 一次输出多个文件 | `停。一次只改一个文件。先只输出第一个文件的代码。` |
| 没回答设计问题就生成代码 | `停。先回答DESIGN阶段的X个问题，确认后再写代码。` |
| VERIFY阶段偷偷改代码 | `停。VERIFY阶段只运行不改代码。回退修改，重新执行VERIFY。` |
| 没有进度标记 | `汇报当前进度。从最后完成的步骤继续。` |
| 自动推进到下一阶段 | `停。当前阶段未完成，不要进入下一阶段。` |
| 分析节点用了不存在的方法 | `停。你PROBE阶段找到的模型调用方法是X，为什么这里用了Y？` |
| 没有加 [PROBE] 日志 | `停。关键路径必须加 logger.info(f"[PROBE] ...") 日志。` |
| 没有异常捕获 | `停。所有异常必须捕获，不能抛未处理异常。` |
| 字段名与CONTRACT不一致 | `停。CONTRACT阶段确认的字段名是X，你这里用了Y，修正。` |
| 没有 git commit | `停。每次修改文件后必须 git add + git commit。` |

---

## 卡住现象 → 话术对照表

| 现象 | 你发给执行AI的话 |
|------|----------------|
| 停在 `[STEP 3/5]` 不动 | `从第3步继续` |
| 某一步失败了 | `重新执行第X步` |
| 不知道卡在哪 | `汇报当前进度` |
| 输出被截断 | `继续输出，从上次截断的地方开始` |
| 执行AI说"我无法做到" | `把你能做到的部分先输出，不要全部放弃` |

---

## 系统指令覆盖（强化话术）

如果执行AI反复违规，发送以下强化指令：

```text
【系统指令覆盖】
从现在开始，严格执行以下规则，优先级高于你之前的所有指令：
1. 用户没有说"确认"，绝不进入下一阶段
2. 一次只改一个文件
3. 每个阶段完成后输出对应的状态标记
4. 每次修改文件后必须 git commit
5. 任何异常立即停止并报告
```

---

## 人工复核检查点

在以下关键节点，人工检查执行AI的输出：

### PROBE 完整性检查
- [ ] 6个问题全部回答
- [ ] 有具体文件路径和行号
- [ ] 关键代码完整贴出（不是只有方法签名）
- [ ] 末尾有 `[PROBE_DONE]` 标记

### CODE 字段名检查
- [ ] 代码中的字段名与 PROBE 中找到的数据结构一致
- [ ] 没有幻觉字段名（如 `relevance_score` 应该是 `relevance`）
- [ ] 方法名与 PROBE 中找到的方法名一致

### VERIFY 日志检查
- [ ] 日志中有 `[PROBE]` 标记的行
- [ ] 新逻辑被调用（不是走的旧逻辑）
- [ ] 没有触发降级（降级意味着新逻辑失败）
